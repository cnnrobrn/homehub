# @homehub/db

The HomeHub database package. Owns every SQL migration, the local
Supabase stack configuration, and the generated TypeScript types that
downstream packages (worker-runtime, web, workers) consume.

Exclusively owned by **@infra-platform**. Other specialists who need a
schema change request it via `tasks/todo.md`.

## What's here

```
packages/db/
├── package.json                 scripts for the local stack + types gen
├── tsconfig.json                extends the repo base; emits to dist/
├── src/
│   ├── index.ts                 public surface: Database type + LOCAL_DATABASE_URL
│   └── types.generated.ts       regenerated by `pnpm db:types`; placeholder until M1
└── supabase/
    ├── config.toml              Supabase CLI project config (project_id, ports, schemas)
    ├── seed.sql                 local-dev seed (header only — filled in later)
    └── migrations/
        ├── 0001_extensions.sql  pgvector, pgmq, pg_cron, pg_trgm, uuid-ossp
        └── 0002_schemas.sql     app, mem, sync, jobs, audit + role grants
```

The Supabase config lives **inside the package** rather than at the repo
root so every workspace artifact for a given concern is colocated. The
infra-platform briefing pins migrations under `packages/db/`; the stack
config follows them.

## Prerequisites

- Docker Desktop (or compatible daemon) running locally.
- Supabase CLI v2.90+ installed (`brew install supabase/tap/supabase`
  on macOS, or follow
  <https://supabase.com/docs/guides/local-development/cli/getting-started>).
  The CLI is a binary, not an npm dep — do not add it to any
  `package.json`.

## Local workflow

From the repo root, every script is namespaced to this package:

```bash
pnpm --filter @homehub/db db:start    # boot the local stack (Postgres, PostgREST, Studio, etc.)
pnpm --filter @homehub/db db:status   # show ports, API URL, anon/service keys
pnpm --filter @homehub/db db:lint     # run supabase db lint against the local stack
pnpm --filter @homehub/db db:stop     # stop the local stack (data preserved)
```

Open Studio at <http://127.0.0.1:54323>, Postgres at `localhost:54322`
(`postgres/postgres`), PostgREST at <http://127.0.0.1:54321>.

After adding or editing tables, regenerate TypeScript types:

```bash
pnpm --filter @homehub/db db:types
```

That script writes `src/types.generated.ts`. Commit the regenerated
file so consumers can typecheck without a running stack.

## Adding a migration

Two canonical flows — pick the one that matches the change.

### Flow A — write SQL directly (preferred for small, reviewed changes)

1. Create the next file in sequence under `supabase/migrations/` using
   the naming convention `NNNN_snake_case.sql` (4-digit prefix, short
   snake-case name). Examples: `0003_app_household.sql`,
   `0011_mem_node_indexes.sql`.
2. Prepend the header block below so every migration is self-documenting.
3. Write idempotent DDL (`create … if not exists`, `create or replace`,
   etc.).
4. Run the stack and apply locally:
   ```bash
   pnpm --filter @homehub/db db:reset:local   # WIPES local data, replays all migrations
   pnpm --filter @homehub/db db:lint
   ```
5. Regenerate types if the migration added or altered tables.

### Flow B — diff against a running stack

1. Start the stack and mutate the schema via Studio or psql.
2. Capture the diff into a named migration file:
   ```bash
   cd packages/db
   supabase db diff -f 0003_short_name   # writes migrations/0003_short_name.sql
   ```
3. Prepend the header block, review and tidy the generated SQL, then
   apply via Flow A step 4.

### Migration header block

Every migration must start with:

```sql
-- Migration: NNNN_short_name.sql
-- Authored: YYYY-MM-DD (ISO-8601)
-- Purpose: one-sentence summary of what this migration does.
-- Owner: @infra-platform
-- Spec: specs/NN-section/file.md (the spec that motivates the change)
```

### Migration numbering

- 4-digit, zero-padded, strictly monotonic.
- No gaps. Never renumber or renumber-and-force.
- Never edit a merged migration — fix forward with a new migration.

## Forward-only, no `supabase db reset` on staging/prod

- `pnpm db:reset` is intentionally disabled — it prints an error and
  exits non-zero. Use `pnpm db:reset:local` for local development only.
- `supabase db reset` against staging or production is **never** run. If
  the staging schema drifts or a migration is wrong, ship a
  forward-fixing migration. Destructive column or table drops are split
  across two migrations (deprecate in release N, drop in release N+1)
  per `specs/10-operations/deployment.md`.

## Push to staging / production

`pnpm db:push` is intentionally absent. Pushing migrations to staging
or production is a **coordinator action** run through the deploy
pipeline (staging auto-applies on merge; production is a manual
promote). See `specs/10-operations/deployment.md`.

## CI

Every PR runs `supabase db lint` against a freshly-started local
Supabase stack in CI (the `migration-lint` job in
`.github/workflows/ci.yml`). If migration lint becomes too slow we will
fall back to a static linter (squawk) — but for now we catch real
issues end-to-end.

## Non-goals for this package

- No Supabase client instantiation. That belongs to
  `packages/worker-runtime` (M0-C) so each caller picks the right
  role (anon, authenticated, service_role) at construction time.
- No RLS policies yet. Policies arrive in M1 alongside the first
  `app.*` tables, per `specs/02-data-model/row-level-security.md`.
